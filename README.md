# 面向端侧大模型低比特量化推理的近似矩阵乘法加速单元设计

## 📌 项目简介
本项目旨在为边缘计算设备上的大语言模型设计高效的硬件加速单元。结合**低比特量化算法**与**近似矩阵乘法**，在保证推理精度的前提下，优化底层硬件的功耗（Power）、面积（Area）与延迟（Performance）。

---

## 🛠️ 环境配置与运行指南（核心速查）

本项目包含“算法验证”与“硬件生成”两大部分，为了防止依赖冲突，部署在两个完全独立的 Conda 虚拟环境中。**每次登录服务器跑代码前，请务必先激活对应的环境！**

### 1. 算法端：低比特量化验证 (Python + PyTorch)
* **环境管家**：Conda
* **环境名称**：`quant`
* **技术栈**：Python 3.10 + PyTorch (适配 CUDA 12.1)
* **激活命令**：
    ```bash
    conda activate quant
    ```
* **⚠️ 运行注意事项 (GPU 资源保护)**：
    ```bash
    CUDA_VISIBLE_DEVICES=0或1 python 你的脚本名称.py
    ```

### 2. 硬件端：加速单元底层设计 (Scala + SpinalHDL)
* **环境管家**：Conda (conda-forge 源)
* **环境名称**：`hw_env`
* **技术栈**：Java 17 (OpenJDK) + SBT
* **激活命令**：
    ```bash
    conda activate hw_env
    ```
* **硬件生成命令**：
    进入对应的硬件工程目录后，呼叫 SBT 管家将 Scala 架构代码翻译成 Verilog 底层芯片代码：
    ```bash
    cd spinal_test
    sbt "runMain MyAdderVerilog"
    ```

## [2026-04-06] 更新：规范 GPU 显卡调用与 PyTorch 动态量化深度验证

### 踩坑记录与核心知识点总结 (💡 必读)
在本次验证中，我们总结了以下关键技术细节：
* **伪造数据测试法 (Dummy Data)**：代码中使用 `torch.randn(1, 4096)` 生成随机张量进行速度测试。在不关心模型输出准确率，只测试硬件推理延迟时，这是最轻量且标准的做法。
* **推理加速关键 `torch.no_grad()`**：在测速代码中必须包裹此上下文管理器，关闭 PyTorch 的自动求导机制，避免在推理时记录计算图，从而大幅降低内存占用并提升速度。
* **Linux 基础操作**：使用 `touch filename` 命令可以快速在服务器创建空白的脚本文件。
* **线性层规模对量化的影响**：`nn.Linear(4096, 4096)` 这种庞大的全连接层是模型体积膨胀的根源（约 1677 万个参数）。量化对这种大型矩阵的存储节省最为显著。
* **异常的速度反转现象 (CPU vs GPU)**：测试结果显示 INT8 量化后速度反而变慢。这证实了 PyTorch 原生 `torch.qint8` 动态量化缺乏 GPU 底层算子支持，导致运行时产生高昂的类型转换开销。这也证明了纯靠上层软件框架实现量化无法发挥最大效能。
* **端侧量化的硬件意义**：虽然体积仅压缩 4 倍，但这对端侧设备意义重大。不仅打破了受限的“内存带宽墙”，并且在底层定制硬件中，INT8 MAC（乘累加单元）的功耗和面积通常不到 FP32 MAC 的十分之一。这也构成了我们后续使用硬件描述语言（Scala/SpinalHDL）设计底层专用加速单元的核心逻辑。
