# Gemini 高效协作与 Token 优化指南

## 核心原则
* [cite_start]**Token 是唯一标准**：限制的是词元数量而非消息条数 [cite: 8, 120]。
* [cite_start]**历史重复读取**：每一条新消息都会导致系统重新读取整个对话历史，造成成本平方级增长 [cite: 29, 33]。

---

## 🚀 提问习惯优化
* [cite_start]**编辑原始消息**：若理解有误，点击“编辑”按钮修改原提示词，而不是发送新的纠错消息 [cite: 42]。
* [cite_start]**合并任务发送**：将多个相关问题放在一个提示词中，避免多次加载上下文 [cite: 58, 70]。
* [cite_start]**15-20 条重置原则**：对话过长会导致 98.5% 的 Token 浪费在读取历史记录上 [cite: 47][cite_start]。建议每 15-20 条消息总结后开启新对话 [cite: 51, 54]。

## 🛠️ 功能与资源管理
* [cite_start]**设置全局偏好**：在“记忆和用户设置”中保存角色背景，减少每轮对话的重复设置 [cite: 83]。
* [cite_start]**利用项目缓存**：上传常用文件至“项目（Projects）”中，避免在不同对话中重复对文档进行 Token 化处理 [cite: 74, 75]。
* [cite_start]**关闭冗余工具**：若不需联网，关闭“网页搜索”；默认关闭“深度思考”模式以节省配额 [cite: 86, 87]。

## 📊 模型与时段调度
* **按需选配模型**：
    * [cite_start]**Haiku 等级**：处理语法检查、简单翻译及 Linux 命令 [cite: 90, 95]。
    * [cite_start]**Sonnet 等级**：处理日常核心开发和论文草稿 [cite: 96]。
    * [cite_start]**Opus 等级**：处理复杂算法推导或深度 Debug [cite: 97]。
* [cite_start]**避开高峰时段**：北京时间 20:00 至凌晨 02:00 额度消耗可能更快，建议分散任务至全天 [cite: 105, 109]。
