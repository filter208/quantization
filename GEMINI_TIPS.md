
Gemini 高效使用与 Token 节省指南本指南旨在通过优化交互习惯，提升 Gemini 在处理代码（Verilog/Scala/Python）及学术任务时的响应质量，并规避使用配额限制 。核心原则：Token 是唯一标准Gemini 限制的是 Token（词元/字符） 数量而非消息条数 。每一条新消息都会导致系统重新读取整个对话历史，造成成本叠加 。🚀 提问习惯优化多编辑，少追问：当模型理解有误时，点击原消息的“编辑”按钮进行修改并重新生成，而不是发送“不对，我的意思是...” 。这样可以替换无效历史，防止对话历史像滚雪球一样膨胀 。任务合并发：将相关的多个任务（如：总结论文、提取要点、翻译）合并为一条指令发送 。这能减少上下文加载次数，让模型一次性看到完整需求，提高代码逻辑的一致性 。15-20 条原则：长对话中绝大部分 Token 都浪费在重读历史记录上 。建议每 15-20 条消息就开启新对话 。迁移技巧：让模型先总结当前对话要点，复制总结并粘贴到新对话的第一条消息中作为背景 。🛠️ 功能与资源管理利用全局偏好：在设置中配置你的角色（如：集成电路专家）和习惯风格（如：偏好简洁代码、中文回复） 。避免在每次开启新对话时重复输入背景信息，节省首轮 Token 消耗 。善用文件上传与缓存：将常用的规格书或长论文上传至项目/对话中。上传一次后系统会进行缓存，后续引用不再重复消耗大量 Token 。精简运行模式：如果不需要实时信息，关闭“网页搜索”等连接器 。默认关闭“深度思考”模式，仅在处理极其复杂的硬件架构或算法推导时开启 。📊 模型与时段调度按需选配模型：Flash (Haiku 等级)：处理代码语法检查、简单翻译、日常 Linux 命令咨询，成本最低 。Pro (Sonnet 等级)：日常核心开发工作，如逻辑设计和论文草稿 。Ultra (Opus 等级)：处理复杂的近似矩阵乘法推导或底层硬件 Debug 。避开高峰时段：AI 服务的额度消耗在高峰期（如北京时间 20:00 至次日 02:00）可能更为严格 。尽量在非高峰时段处理资源密集型任务（如大代码量重构或超长文档分析） 。
